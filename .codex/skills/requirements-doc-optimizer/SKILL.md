# Requirements Doc Optimizer

## When To Use

Use this skill when the user asks to:

- 优化、改写、重整需求文档
- 把 Axure 导出的需求页整理成更清晰的 HTML/HTM 文档
- 将原文档中的原图与功能描述一一对应
- 修正截图缺失、截图不完整、裁剪错误、图文不匹配
- 按页面从上到下的顺序重组模块
- 保留原始需求内容，但改成更清晰的展示结构

Typical inputs:

- `requirments/` 下的 Axure 导出 `.html` / `.htm`
- 同目录或同功能文件夹下的 `images/`、`files/`、`resources/`
- 用户已经整理过但仍需校正图片或结构的中间版文档

## Non-Negotiable Rules

1. 必须优先使用原始需求文档里的原图，不要自己画图，不要用替代图。
2. 图片必须和功能描述一一对应，不能只放“相近页面图”。
3. 文档结构必须优先按页面视觉顺序组织，默认从上到下拆模块。
4. 模块命名优先使用真实页面区域/元素，如：头图、入口、弹窗、列表、空态、底部栏。
5. 不要先按抽象规则分组，例如“规则说明”“功能汇总”“逻辑校验”，除非页面本身没有稳定结构。
6. 一个模块里如果有多个独立视觉点，必须拆成多张原图分别对应，不要强行用一张图概括。
7. 图片下方说明文案不能遮挡原图有效信息。
8. 不要漏需求；每个模块都要保留对应的需求描述或需求摘录。
9. 默认新文档另起名字保存，不覆盖原始需求文档；只有用户明确要求时才原地改。
10. 如果用户明确指出某一版是“最好的一版”，优先保留那一版的排版结构，只修正图片、映射关系和遗漏内容。

## Default Output Pattern

默认把需求文档整理成：

- 页面总览 / 整理说明
- 按页面从上到下的模块区块
- 每个模块包含：
  - 模块标题与简述
  - 对应原图，必要时为多张
  - 图中对应关系说明
  - 功能点拆解
  - 原始需求摘录

默认图文关系：

- 左侧或上方放原图
- 右侧或下方放功能描述
- 多张图时按 `原稿图1 / 原稿图2 / 原稿图3` 标注
- 描述中的编号必须能回指到具体图片或图片中的具体区域

## Workflow

### 1. Locate The True Source

- 优先使用用户给的原始需求文件路径。
- 如果当前目录只有单个 HTML、缺少 `images/`、`files/`、`resources/`，不要直接猜图。
- 需要先找到完整 Axure 导出目录，再从完整导出中提取原图或渲染页面。
- 如果用户已经给了原图目录，优先从该目录取图。

### 2. Read The Page In Visual Order

- 先识别页面从上到下有哪些真实区域。
- 再识别每个区域里的主要元素、弹窗、状态、结果页或分享页。
- 按“页面看到的顺序”建模块骨架，不按抽象规则先分类。

优先顺序：

- 页面入口
- 顶部信息区
- 核心弹窗/主列表
- 空态/异常态
- 结果页主体
- 底部功能区
- 分享/咨询/反馈等衍生页面

### 3. Match Images Before Writing

- 先确认每个模块到底对应哪一张原图，再开始写文档。
- 一张图不够时，拆成两张或多张。
- 如果用户特别指出“某入口没有对应截图”，必须单独补图。
- 如果图是从长页面中裁出来的，必须检查裁剪是否完整、是否截到了真正功能点。

图片匹配标准：

- 图里真的能看到该功能
- 图的状态和描述一致
- 不能拿 A 模块的图去解释 B 模块
- 不能用空态图去代表正常态
- 不能用普通结果页去代表分享结果页

### 4. Export And Save Original Images

- 默认把导出的原图保存到功能目录下的 `image/` 目录。
- 文件名要稳定、可读、可复用，例如：
  - `entry-share.png`
  - `entry-pk.png`
  - `selector.png`
  - `empty.png`
  - `share-result.png`
- 如果已有正确原图，直接复用，不重复生成。
- 如果需要从 Axure 页面裁图，先校正坐标，再批量导出。

### 5. Rewrite By Module

- 每个模块都要写清楚“图里是什么”和“这个图对应什么功能”。
- 需求点要挂在具体页面元素下面，不单独漂浮成一段规则。
- 文案应是需求语言的整理版，不要脱离原意。

推荐顺序：

1. 模块标题
2. 模块简述
3. 原稿对应图
4. 图中对应顺序
5. 功能点拆解
6. 原始需求摘录

### 6. Save As A New Optimized File

- 默认另起名字保存，例如：
  - `xxx需求整理.htm`
  - `xxx-原图对照重写版.html`
  - `xxx-改写-入口重整版.html`
- 文件名要体现“整理/改写/原图对照”。
- 不要覆盖用户原始文档。

### 7. Validate Before Finishing

交付前逐项检查：

- 所有图片都能正常显示
- 图片和描述一一对应
- 线路入口、分享入口、空态、分享页等容易漏的模块是否都补齐
- 有没有图片裁剪错误
- 有没有说明文案压住原图有效信息
- 有没有遗漏原始需求点
- 模块顺序是否真的是按页面从上到下
- 新文档是否另存

## Writing Guidance

- 默认用中文输出。
- 保留原需求中的关键文案、提示语、按钮名、标题名。
- 可以改写表达，让结构更清晰，但不要改业务含义。
- 一个模块里如果既有“正常态”又有“空态/弹窗/分享态”，要拆开写，不要混在一张图里硬讲。
- 如果原稿右侧有标注说明，可以保留其价值，但文档主体仍以“页面模块化整理”表达为主。
- 若用户特别要求“参照某一版最好版本继续调整”，优先延续该版的视觉结构和版式。

## Quality Checklist

- [ ] 使用的是原始来源图片，不是替代图
- [ ] 每张图都和当前模块描述直接对应
- [ ] 页面模块顺序符合从上到下的阅读顺序
- [ ] 多视觉点模块已拆成多张图
- [ ] 活动选择弹窗、空态图、分享页、入口图没有互相错放
- [ ] 图片未被说明文案遮挡
- [ ] 原始需求摘录未遗漏关键规则
- [ ] 新文档已另存，不覆盖源文档

## Example Triggers

- “帮我优化这个需求文档”
- “把这个 Axure 需求页重新整理一下”
- “用原图重写需求文档，图片和描述一一对应”
- “按页面从上到下重新拆模块”
- “把截图都替换成原文档里的正确图片”
