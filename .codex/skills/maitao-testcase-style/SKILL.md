# 麦淘内部测试用例风格指南（XMind优先）

## 使用时机
- 当用户要求“按照我的风格”“按照麦淘旅行家用例风格”“按脑图/XMind风格”编写或更新测试用例时。
- 当需求文档来自 `requirments/+麦淘旅行家测试用例（落地页）.md`、同风格脑图导出稿，或仓库内已有同风格 testcase 文件时。
- 当 `requirements-testcase-writer` 已负责通用“从需求生成用例”的流程，而当前任务还需要继续沿用麦淘既有措辞、颗粒度、模块顺序和 XMind 分支节奏时。
- 若用户只需标准表格或西式模板，请勿启用本技能。

## 默认交付格式
1. 默认最终交付物是 `testcases/` 下的真实 `.xmind` 文件，不是表格，也不是普通 Markdown。
2. 除非用户明确要求查看 Markdown 草稿，否则不要创建、更新、同步任何 `.md` 测试用例文件。
3. XMind 默认结构为：`根节点 -> +页面/模块节点 -> +功能节点 -> 用例节点 -> 步骤内容节点 -> 预期结果内容节点`。
4. 真实 `.xmind` 中，不额外创建名为“测试步骤”或“预期结果”的独立节点，直接把编号文本写入对应子节点。
5. 画布默认沿用仓库既有风格：`逻辑图`、`晨曦`、`彩虹分支开启`、`同级主题对齐开启`。

## 写作流程
1. **收集信息**：用 `rg "模块名" requirments/+麦淘旅行家测试用例（落地页）.md` 定位对应内容；必要时同步查阅其它需求文件。
2. **划分模块**：先按页面元素和页面区域搭骨架，再按页面从上到下顺序列出模块与子节点，确认是否需要新增二级或三级标题。
3. **确定交付形态**：默认直接按 XMind 节点组织并产出 `.xmind` 文件，不再同步 Markdown 版本。
4. **确定 case 粒度**：默认先尝试“一个页面模块写成一个完整 case”；只有当模块内部场景明显独立、且拆开更清楚时，才拆成多个 sibling case。
5. **挂载规则**：把需求描述里的展示规则、交互规则、状态规则、异常规则，放到它实际发生的页面元素下面，不单独抽成脱离页面的规则模块。
6. **展开步骤**：写清楚入口、前置条件、操作路径，必要时拆分不同端或账号态。
7. **描述预期**：先写页面元素，再写默认展示，再写切换交互，再写异常/边界；对多分支情况使用 `a.` `b.` `c.` `d.` 分项罗列。
8. **主动补交互**：如果页面里有 Tab、品类标签、线路/套餐、筛选、展开收起、关闭重开、登录跳转、底部栏联动，默认都要考虑是否写进同一个 case。
9. **串联编号**：客户端用例编号沿用 `PK-C-xxx`，埋点 `PK-A-xxx`，后台 `PK-B-xxx`，放在 `###` 标题或预期首条。
10. **互锁校验**：逐条比对原需求，确保没有漏掉模块/子条件；如需求含“见用例《xxx》”需补充跳转链接或引用说明。

## 文本风格
- 中文标点优先，保持“，”“：”“（）”等写法。
- 数字或字母编号与原文一致（`1-d-1）` 等结构不要改写）。
- 语气保持客观陈述，例如“显示”“提示”“进入”。
- 涉及金额、时间、链接，要抄原文，不自行推测。
- 涉及仓库通用业务口径，例如 `预订产品价格设置`、起价匹配、日期价格展示等，沿用 `requirements-testcase-writer` 中定义的默认业务解释；本技能不重复定义业务规则。
- 默认不要先按抽象功能点或规则点分组；优先写成“页面区域/元素 -> 场景”。
- 模块命名贴近页面真实元素，例如 `头图`、`悬浮按钮`、`弹窗`、`列表`、`卡片`、`底部栏`，避免泛化命名如 `规则校验`、`功能模块`、`列表逻辑`。
- 如果用户偏好“大 case”，优先把一个模块内的显示逻辑、切换交互、边界情况收敛到同一个 `###` 用例标题下，再在预期里用 `a.` `b.` `c.` `d.` 展开。
- 对标签、角标、按钮状态、字段显隐这类多条件逻辑，不要只写一句“按规则显示”，要把关键组合列出来。
- 写卡片、弹窗、表格、底部栏时，默认先检查是否遗漏页面元素描写，例如头图、标题、标签、评分、价格、按钮、关闭按钮、已选态、空态按钮。

## 常见反例
- 反例 1：一个弹窗模块被拆成十几个很碎的 case，每个 case 只写一句结论。
  正确方向：先收成 1~3 个大 case，再在预期里拆 `a.` `b.` `c.` `d.`。
- 反例 2：只写“列表按规则展示”“标签按规则显示”，没有写列表里到底有什么元素、标签在什么条件下显示或不显示。
  正确方向：先写页面元素，再列条件组合。
- 反例 3：只写默认态，不写切换 Tab、切换标签、切换线路/套餐后的变化。
  正确方向：把切换前后差异写进同一个 case。
- 反例 4：模块名写成 `规则校验`、`功能汇总`，看不出这是页面上的哪个区域。
  正确方向：改成真实页面元素名，如 `选择活动弹窗`、`产品卡片`、`底部栏`。
- 反例 5：把需求原文直接拆行塞进预期，没有转成可观察的结果。
  正确方向：改成“显示什么、点击后怎样、切换后怎样、异常时提示什么”。

## 收拆判断
- 优先收成一个大 case：
  - 同一模块里的元素、默认态、切换态、边界态彼此强关联
  - 用户偏好旧脑图那种“大 case + 多分支预期”
  - 拆开后会造成重复步骤和重复前置条件
- 优先拆成多个 case：
  - 同一模块内存在几个明显独立的交互目标
  - 某个分支需要完全不同的入口、数据准备或预期结构
  - 用户明确要求更细颗粒度
- 默认先收后拆，不要先碎写再尝试合并。

## 质量检查清单
- [ ] 若用户未明确要求其它格式，最终交付优先为 `.xmind`。
- [ ] 未额外生成或同步 `.md` 测试用例文件。
- [ ] XMind 节点结构符合“页面/模块 -> 功能 -> 用例 -> 步骤 -> 预期”的层次，不回退成表格思路。
- [ ] 每个模块至少有一个 `###` 节点。
- [ ] 模块顺序符合页面从上到下的阅读/操作顺序。
- [ ] 用例先挂在页面元素下，再展开正常、异常、边界场景。
- [ ] 每个重点模块都检查过是否更适合写成“一个完整 case + 多分支预期”。
- [ ] 预期结果里先有页面元素描写，再有交互变化和条件分支。
- [ ] Tab切换、标签切换、线路/套餐切换、展开收起、关闭重开等交互没有遗漏。
- [ ] 标签/角标/显隐逻辑已用 `a.` `b.` `c.` `d.` 列出关键组合。
- [ ] 步骤和预期都写全，且步骤与预期数量、顺序匹配。
- [ ] 涵盖正常、异常、空态、埋点/后台等多维视角。
- [ ] 引用其它用例或静态页时，写明名称或链接，以便交叉追踪。
