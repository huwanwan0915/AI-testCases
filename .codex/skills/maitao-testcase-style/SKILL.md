# 麦淘旅行家测试用例风格指南

## 使用时机
- 当用户要求“按照麦淘旅行家用例风格”编写/更新测试用例时。
- 当需求文档来自 `requirments/+麦淘旅行家测试用例（落地页）.md` 或风格相同的脑图导出稿时。
- 若用户只需标准表格或西式模板，请勿启用本技能。

## 输出骨架
1. `#` 顶层标题写产品/项目名称，可包含“+”前缀（与脑图相同）。
2. `##` 表示一级模块（页面/流程），顺序按需求文件出现顺序。
3. `###` 表示具体用例主题，命名推荐为“模块-场景”；若需要保留编号，可写成“PK-C-001 线路PK按钮展示”。
4. `####` 行写测试路径首句（如 `#### 1. 准备环境...`），后续步骤直接换行继续以“2.”、“3.”开头。
5. 步骤区和结果区之间空一行。
6. 预期结果使用“两空格+短横线”缩进（`␠␠-`），保持与落地页文档相同的嵌套形态，可在条目前继续沿用“1.”、“a.” 等编号。
7. 需要补充的细则（如分端差异、跳转链接、提示语）直接写在预期区域，保持原文案。
8. 如需备注或额外引用，用 `>` 创建引用块。

## 写作流程
1. **收集信息**：用 `rg "模块名" requirments/+麦淘旅行家测试用例（落地页）.md` 定位对应内容；必要时同步查阅其它需求文件。
2. **划分模块**：按需求顺序列出模块与子节点，确认是否需要新增二级或三级标题。
3. **展开步骤**：写清楚入口、前置条件、操作路径，必要时拆分不同端或账号态。
4. **描述预期**：覆盖页面展示、交互响应、跳转、异常提示、埋点等；对多分支情况使用 `a.` `b.` 分项罗列。
5. **串联编号**：客户端用例编号沿用 `PK-C-xxx`，埋点 `PK-A-xxx`，后台 `PK-B-xxx`，放在 `###` 标题或预期首条。
6. **互锁校验**：逐条比对原需求，确保没有漏掉模块/子条件；如需求含“见用例《xxx》”需补充跳转链接或引用说明。

## 文本风格
- 中文标点优先，保持“，”“：”“（）”等写法。
- 数字或字母编号与原文一致（`1-d-1）` 等结构不要改写）。
- 语气保持客观陈述，例如“显示”“提示”“进入”。
- 涉及金额、时间、链接，要抄原文，不自行推测。

## 质量检查清单
- [ ] 每个模块至少有一个 `###` 节点。
- [ ] 步骤和预期都写全，且步骤与预期数量、顺序匹配。
- [ ] 涵盖正常、异常、空态、埋点/后台等多维视角。
- [ ] 引用其它用例或静态页时，写明名称或链接，以便交叉追踪。
- [ ] 输出为 UTF-8 Markdown，无额外表格。
